Incharj — Core Architecture

Connectors · Indexer · Search algorithm · How data flows from source to result

1

Connector — Fetching documents from external sources

OAuth 2.0 integrations that stream document metadata and content into the indexing pipeline

Step 1
OAuth Setup
›User clicks Connect → frontend calls getOAuthUrl()
›OAuth state param stored in localStorage keyed as oauth_state:{state}
›User approves at provider → redirect to /oauth/:kind/callback
›Code exchanged for tokens → encrypted with AES-GCM → stored in connectors.credentials
Step 2
listDocuments()
›Async generator — yields one ConnectorDocument at a time
›Incremental: reads config.last_synced_at (ISO string) to filter only changed docs at the source API
›Google Drive: modifiedTime > last_synced_at query param
›Notion: filter.last_edited_time.after
›Respects config.max_documents cap (testing)
Step 3
fetchContent()
›Called per document — separate from listing so rate limits can be managed independently
›Google Docs → exported as plain text via Drive API
›Sheets → exported as CSV
›PDF → streamed as Buffer → parsed with pdf-parse (CJS require())
›HTML → tags stripped before indexing
›Max 2 MB per file — stream capped to avoid OOM
Step 4
Credential Refresh
›refreshCredentials() called at sync start before listing
›Google: exchanges refresh token for new access token when expired
›Updated credentials re-encrypted and written back after sync completes
›Notion / Slack: long-lived tokens, no refresh needed
2

Indexer — Transforming content into searchable records

Each document runs through ingestDocument() in its own PostgreSQL transaction — one failure never aborts the sync

1 · Hash
Deduplication
›Compute SHA-256("${title}::${content}")
›Title is included so a rename-only change still triggers re-index
›Compare against documents.content_hash
›Hash match → return "skipped" immediately
✓Incremental syncs skip the majority of documents
2 · Chunk
Text Chunking
›chunkText(content) in utils/chunker.ts
›Chunk size: 800 characters
›Overlap: 100 characters between adjacent chunks
›Overlap preserves context across chunk boundaries so search doesn't miss phrases that straddle a split
›Approximate token count stored per chunk
3 · Store
Upsert to DB
›Upsert documents row on (connector_id, external_id)
›Delete all old document_chunks for the doc
›Insert new chunks with chunk_index, content, token_count
›All inside a single BEGIN / COMMIT — atomic per document
4 · Index
Search Vector
›After upsert: UPDATE documents SET search_vector = to_tsvector('english', title)
›Each chunk: search_vector = to_tsvector('english', content) on insert
›Both columns have GIN indexes — no runtime to_tsvector() at query time
✓Pre-computation is why search is fast even at scale

Full-Text Search — How ranking works
-- 1. Parse the query into a tsquery websearch_to_tsquery('english', 'product roadmap') → 'product' & 'roadmap' -- 2. LATERAL join: find best matching chunk per doc -- (pre-filtered by GIN index — only matching chunks) LATERAL ( SELECT dc.content, dc.search_vector FROM document_chunks dc WHERE dc.document_id = d.id AND dc.search_vector @@ tsq.q ← GIN index hit ORDER BY ts_rank_cd(dc.search_vector, tsq.q) DESC LIMIT 1 ) -- 3. Build combined weighted tsvector setweight(d.search_vector, 'A') ← title (most important) || setweight(to_tsvector('english', chunk.content), 'B') -- 4. Score = rank × time decay ts_rank_cd(sv, tsq.q, 32) * exp(-age_seconds / (90 * 86400)) ↑ 90-day half-life decay
ts_rank_cd Cover density ranking — rewards documents where query terms appear close together, not just frequently
weight 'A' Title matches score 4× higher than body text — finding "roadmap" in a title beats finding it buried in content
time decay A 90-day-old doc at equal relevance scores ~37% lower. A year-old doc scores ~2% of an equivalent new one
LATERAL Only the best chunk per doc is ranked — avoids double-counting multi-chunk documents
Snippet generation · ts_headline
input cap Chunk content is capped at 5 KB before passing to ts_headline() — prevents slow processing on large documents
delimiters Matches wrapped in << and >> — frontend highlights in yellow
fragments Up to 2 fragments of 40 words each, joined with …
"…the product roadmap for Q3 focuses on search quality improvements and connector reliability…
…updated the roadmap after the customer feedback session…"
Fuzzy fallback · pg_trgm similarity
trigger Only runs if FTS returns 0 results — catches typos, partial words, and non-English terms
scope Title only — chunk content can't use a GIN index inside a subquery, so it would require a full table scan
threshold similarity(title, q) > 0.1 — low enough to catch roadmpa → roadmap, high enough to filter pure noise
index Uses ix_documents_title_trgm (GIN gin_trgm_ops) — stays fast at scale
QueryMatchesScore
roadmpa"Q3 Roadmap"~0.44

Stop word guard
problem English stop words like the, a, is are stripped by PostgreSQL's English dictionary. websearch_to_tsquery('english', 'the') returns an empty tsquery
old behaviour FTS returns 0 → fuzzy fallback scans every document title → very slow (> 60s on large datasets)
fix Before any query, check websearch_to_tsquery(q)::text = '' → if empty, return immediately with 0 results. No DB scan at all.
Filters & Pagination
FilterSQL
orgd.org_id = $1 (always)
connector_idd.connector_id = $N
kindd.kind = $N
date_fromd.mtime >= $N
date_tod.mtime <= $N
count query Runs in parallel with the main query via Promise.all — no extra round-trip latency
pagination Default limit=20, offset=0. Response includes total for frontend page controls

Database indexes that make it fast

All search queries use pre-built GIN indexes — no runtime tsvector computation

Index summary
Index Table Type Used by
ix_documents_search_vector documents GIN FTS document-level match: d.search_vector @@ tsq
ix_chunks_search_vector document_chunks GIN LATERAL join pre-filter: dc.search_vector @@ tsq
ix_documents_title_trgm documents GIN trgm Fuzzy fallback: similarity(title, q) > 0.1
ix_documents_connector_id documents btree connector_id filter + doc count after sync
ix_document_chunks_document_id document_chunks btree Delete old chunks on re-index
